# マウス・クリップボード・imgcatの仕組み

---

**kawaz:** マウス操作の取得とかもコレでカバーされる? あとクリップボード貼り付けとかimgcatみたいな仕組みはどう動く?

**Claude:**

結論から言うと**3つとも既存のFFIリストに追加不要**です。全部ANSIエスケープシーケンスのレイヤーで完結します。

## マウス操作

Xterm Mouse Protocolという仕組みで、**エスケープシーケンスを送って有効化し、イベントもエスケープシーケンスとしてstdinから受け取る**だけです。

```
// 有効化（pure MoonBitで書ける）
\x1b[?1000h   // クリックのみ
\x1b[?1002h   // ドラッグも
\x1b[?1003h   // 全マウス移動
\x1b[?1006h   // SGR拡張モード（座標が大きい時必要）

// 受信イベントの形式（stdinから来る）
\x1b[<0;10;5M   // ボタン0、x=10、y=5、押下
\x1b[<0;10;5m   // 同、リリース
```

TTYのraw modeが取れていれば、あとはstdinのパースだけ。pure MoonBitで実装できます。

## クリップボード貼り付け

2種類あります。

**Bracketed Paste Mode（受け取り側）**

```
\x1b[?2004h   // 有効化
// ペースト時にターミナルが自動でこれを付けて送ってくる
\x1b[200~     // ペースト開始
（ペースト内容）
\x1b[201~     // ペースト終了
```

改行を含むペーストと通常のEnterキーを区別できるようになる。

**OSC 52（クリップボードへの書き込み）**

```
\x1b]52;c;<base64エンコードされたテキスト>\x07
```

ターミナルエミュレータがこれを受け取るとOSのクリップボードに書き込む。読み取りも同様にOSC 52で要求できます（ターミナルが対応していれば）。

## imgcat（iTerm2 / Kitty Graphics Protocol）

どちらも「ターミナルエミュレータが解釈する特殊シーケンスをPTYに流す」だけです。

**iTerm2 (OSC 1337)**
```
\x1b]1337;File=width=100px;height=50px;inline=1:<base64画像データ>\x07
```

**Kitty Graphics Protocol (APC)**
```
\x1bGa=T,f=32,s=幅,v=高さ;<base64チャンク>\x1b\
```

**shimuxにとって何が嬉しいか**

shimuxはPTYを透過するだけなので、アプリが流すこれらのシーケンスは**何もしなくても自動的にGhosttyに届きます**。ただしログバッファに記録するときはバイナリが大量に混じるので、ログ用のHookでOSC/APCシーケンスを検出してスキップするか圧縮する処理が必要になります。

## pure MoonBitで書けるレイヤー

```
[FFI (Rust)]
  PTY, socket, signal, termios  ← 前回のリスト

[pure MoonBit]
  ANSIエスケープ生成・パース     ← マウス有効化、OSC 52など
  Mouse Protocolパーサ
  Bracketed Pasteパーサ
  OSC/APCシーケンス検出（ログフィルタ用）
  Hookチェーン
```

マウスとクリップボードはむしろMoonBitの設計の見せ所になる部分です。
